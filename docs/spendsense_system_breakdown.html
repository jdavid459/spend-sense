Portfolio-ready SpendSense walkthrough

SpendSense: system breakdown, setup guide, and architecture tutorial

SpendSense is a personal spend analytics project built to show strong analytics engineering, data engineering, and applied AI fundamentals. It takes raw Chase CSV data, models it in dbt on top of DuckDB, serves analytics through Dash, and uses Cohere for enrichment, search, and grounded summaries. The project is intentionally designed to be easy to explain in interviews because the layers, ownership boundaries, and metric definitions are explicit.

1) What SpendSense is

Input

Raw Chase credit card CSVs in either demo or private mode.

Core engine

Python ingests the CSV into DuckDB, then dbt transforms it into analytics-ready models.

User experience

A Dash app reads the modeled marts and adds AI-powered search and summaries.

Main idea: this is not just a dashboard. It is a small analytics product with ingestion, modeling, governed marts, metric definitions, and AI features layered on top.

2) Quick setup

Use this path if you want to reproduce the project locally.

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env

# Optional: generate demo data
python scripts/generate_demo_data.py

# Load CSV into DuckDB raw table
python scripts/ingest_chase_csv.py

# Build dbt models
cd dbt && dbt deps --profiles-dir . && dbt build --profiles-dir .
cd ..

# Launch dashboard
python app/app.py

Demo mode

Safe for sharing. Uses the synthetic CSV at data/demo/chase_transactions_demo.csv.

Private mode

Uses a local Chase export in data/private/. Private files are gitignored.

Key config lives in .env.example and src/config.py.

3) End-to-end architecture

SpendSense separates deterministic analytics engineering from AI enhancement. dbt owns the core business logic; Cohere adds enrichment, retrieval, and narrative layers on top.
Step 1

Source data

Chase CSV or synthetic demo CSV.

Step 2

Python ingestion

Loads rows into raw.chase_transactions in DuckDB.

Step 3

dbt transforms

Builds staging, intermediate, and mart layers.

Step 4

Analytics layer

Reusable transaction facts, recurring detection, anomalies, KPI marts, and metrics.

Step 5

Dash app

Reads dbt marts into an interactive dashboard.

Step 6

Cohere services

Merchant enrichment, grounded summaries, embeddings, rerank, and merchant semantic profiles.

Design philosophy: most business logic lives in dbt/SQL. Python stays thinner and handles ingestion, app behavior, and AI API workflows.

4) How the pipeline works

Stage What happens Relevant files
Input selection The app decides whether to use demo or private data, and private mode can auto-detect the latest CSV. src/config.py
Demo generation Synthetic data is generated with recurring charges, anomalies, inconsistent merchants, and duplicates. scripts/generate_demo_data.py
Ingestion Python validates expected Chase columns, adds lineage fields like source row number and row hash, then loads the raw table into DuckDB. scripts/ingest_chase_csv.py
dbt build dbt transforms the raw table into clean staging models, feature models, and business-facing marts. dbt/
Dashboard load Dash queries the marts from DuckDB and applies filters for date, category, merchant, and view mode. app/app.py
AI layer Cohere calls enrich merchants, generate summaries, and power semantic transaction search. Results are cached in DuckDB. src/

5) How dbt is used

dbt is the modeling backbone of SpendSense.

Project structure

dbt is configured with separate staging, intermediate, and marts schemas to make the pipeline easy to explain.

See: dbt/dbt_project.yml, dbt/profiles.yml, generate_schema_name.sql

Why it matters

dbt makes the transformation layer versioned, testable, documented, and reusable outside the dashboard.

Layer Purpose Examples
Staging Standardize raw Chase fields and prepare clean transaction columns. stg_chase_transactions.sql
Intermediate Apply merchant normalization, enrichment precedence, and transaction-level features. int_merchant_normalization.sql
int_transaction_features.sql
int_recurring_transactions.sql
Marts Expose dashboard-ready facts, metrics, anomaly marts, recurring marts, and data quality outputs. dbt/models/marts/
Analytics engineering highlight: the project explicitly models denominator choices. Spend shares are based on debit spend, while credits/payments are separated so metrics stay correct.

6) How the data is modeled

Core model flow

raw.chase_transactions
  -> staging.stg_chase_transactions
  -> intermediate.int_merchant_normalization
  -> intermediate.int_transaction_features
  -> intermediate.int_recurring_transactions
  -> marts.fct_transactions
  -> marts for anomalies, recurring, merchant review, KPIs, and daily metrics

Main model grains

Model Grain Why it exists
stg_chase_transactions One row per raw CSV transaction Standardized dates, amount fields, debit/credit flags, and transaction ID.
int_merchant_normalization One row per transaction Determines normalized merchant, merchant group, category source, AI confidence, and provenance.
int_transaction_features One row per transaction Adds z-scores, merchant/category comparisons, and spacing between transactions.
fct_transactions One row per transaction Final fact table consumed by the app.
mart_merchant_review One row per raw description / merchant combination Prioritizes cleanup candidates and shows what still needs governed mapping.
mart_daily_metric_values One row per metric x day x filterable dimensions Supports rolling 30-day metric views in the dashboard.

Merchant normalization logic

SpendSense uses a clear precedence rule:

seed rule > Cohere cache (confidence >= 0.70) > raw fallback

This is strong from a governance perspective because AI does not silently override deterministic logic. The source of each classification is preserved using merchant_source and category_source.

Metric modeling

Important marts include:

  • mart_spend_kpis.sql
  • mart_monthly_kpis.sql
  • mart_category_metrics.sql
  • mart_data_quality.sql
  • mart_metric_summary.sql
Analytics engineering highlight: this metrics layer is interview-relevant because it shows reusable marts, rolling windows, concentration metrics, volatility, anomaly exposure, and AI coverage tracking.

7) How data flows into Python and Dash

DuckDB access

src/db.py provides a simple query helper that reads modeled tables from DuckDB into pandas DataFrames.

App orchestration

app/app.py loads marts, applies filters, builds KPI cards, and renders tabs for overview, anomalies, recurring, metrics, merchant cleanup, and AI summary.

How the app uses the modeled data

Dashboard area Main source Why this is useful
Overview marts.fct_transactions Flexible filtered charts across time, categories, merchants, and spend behavior.
Transactions marts.fct_transactions Base transaction table plus semantic search.
Anomalies marts.mart_anomalies and filtered fact data Explainable outlier review.
Recurring marts.mart_recurring_spend Subscription/recurring burden analysis.
Metrics marts.mart_daily_metric_values Filterable rolling 30-day metrics with trend lines.
Merchant Cleanup marts.mart_merchant_review Operationalizes cleanup work as a governed analytics artifact.

The app also splits more specialized logic into components: app/components/ai_summary.py and app/components/semantic_search.py.

8) How Cohere is used

SpendSense uses Cohere in multiple ways, not just one feature.

Cohere product Used for Relevant code
Chat Merchant enrichment, merchant semantic profiles, and grounded spend summaries. scripts/enrich_merchants.py
src/merchant_profiles.py
src/cohere_client.py
Embed Semantic transaction search using vector embeddings. src/semantic_search.py
scripts/backfill_transaction_embeddings.py
Rerank Improves search precision after semantic retrieval. src/semantic_search.py

Cohere usage patterns in the system

1. Merchant enrichment

Unmapped raw descriptions are sent to Cohere Chat for suggested merchant name, category, merchant group, confidence, and reasoning. The result lands in ai.merchant_enrichment_cache.

2. AI summary

The summary tab first shows a deterministic summary, then optionally calls Cohere for a more natural write-up grounded in dbt metrics and filtered transactions.

3. Merchant semantic profiles

Cohere Chat creates reusable merchant summaries and semantic tags, which improves retrieval quality for search.

4. Semantic search

Transactions are embedded, a query is embedded, similarity is scored, then Cohere Rerank improves final result quality.

AI governance pattern

One of the strongest parts of this design is that AI outputs are treated as data products:

  • cached in DuckDB
  • confidence-scored
  • auditable
  • precedence-controlled
  • measured through coverage and fallback metrics

Cache definitions live in src/ai_cache.py.
